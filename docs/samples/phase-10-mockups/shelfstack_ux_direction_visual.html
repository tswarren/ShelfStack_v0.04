ShelfStack UX Direction

View contracts and interaction standards

Operational Keyboard-first Progressive disclosure

Target experience

Fast, calm, predictable bookstore operations.

ShelfStack should organize complex catalog, product, inventory, order, and POS records into task-centered screens that answer what staff need to know now and what they should do next.

Summary first Details on demand Brand-aware, not noisy Dense, not cramped
Primary rule One most likely next action should be visually obvious.
Context rule Keep users in the current workflow; use modals, drawers, and Turbo updates for bounded edits.
Information rule Summarize related records before exposing full detail.

Design principles

How screens should behave

These principles keep the interface operational rather than decorative and prevent every control from looking equally important.

1

One primary action

Each screen or section should make the next likely action clear. Secondary actions remain visible but quieter.

≠

Not everything is a button

Filters, tabs, status badges, row actions, and destructive controls need distinct treatments.

⌘

Keyboard-friendly

Focus, Enter, Escape, scan fields, and post-save behavior are intentional, not browser-default accidents.

↺

Preserve context

Use focused editors and partial updates so staff do not lose their place in a workflow.

▤

Summarize first

Availability, setup completeness, demand, and related records should be summarized before detail tables dominate.

◎

Calm hierarchy

Spacing, cards, typography, and restrained brand color should reduce overwhelm.

Core view contracts

Each page type has a defined job

A screen should be easy to classify. Once the view type is known, structure, focus, primary action, and feedback behavior should be predictable.

⌕

Index / Search

Find records, narrow a queue, or start a new task.

First focus: Search fieldPrimary: Search / Add
i

Detail / Overview

Explain the record and what can be done now.

First focus: Usually nonePrimary: Continue / Edit
I

Item Overview

Unify catalog, product, SKU, inventory, orders, and demand as one item.

First focus: Item search/actionPrimary: Sell / Receive / Order
✎

Form / Edit

Change one record or coherent setup area.

First focus: Invalid fieldPrimary: Save
→

Workflow

Guide staff through a multi-step operational task.

First focus: Current step inputPrimary: Continue / Complete
$

POS / Register

Prioritize scan focus, cashier speed, tender clarity, and safe completion.

First focus: Scan fieldPrimary: Settle / Complete
▥

Report

Review, filter, print, export, and understand operational results.

First focus: Filter if neededPrimary: Run / Print / Export
⚙

Setup List

Maintain configuration records with denser but consistent controls.

First focus: Search or AddPrimary: Add

UX rules

Use drawers or expanded rows for variant detail. Summarize sales, receiving, and audit history so they do not dominate the page.

Important distinction

Vendor-source warnings should apply only to vendor-orderable variants. Used and buyback variants should not be treated as incomplete because they lack vendor sources.

Framework decision rule

Choose tools only after the UX contract is agreed

RubyUI, DaisyUI, or custom ss-* components should be evaluated against the desired ShelfStack feel, Rails/Turbo/Stimulus fit, keyboard support, brand consistency, and incremental adoption.

Standardized screen checklist

  • Defined view type
  • Clear primary action
  • Distinct secondary/destructive actions
  • Standard field styling
  • Predictable button placement
  • Status badges not styled as buttons
  • Human-readable money/percent inputs
  • Validation near fields
  • Intentional keyboard focus
  • Turbo preserves focus
  • Summaries before details
  • Advanced details disclosed progressively
