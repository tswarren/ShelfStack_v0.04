## Summary

Makes PO vendor quantity lifecycle and receipt posting explicit: vendor buckets on PO lines, authoritative inbound supply math, atomic inbound→on_hand conversion on receipt post (FIFO, split partials, idempotent per receipt line), reverse-FIFO release on shortage/close-short, and bounded PO/receipt UI labels.

## Delivered

### Schema

- `purchase_order_lines`: vendor quantity state, buckets, `quantity_closed_short`, recorded-at/source markers
- `demand_allocations`: conversion traceability FKs; `converted` terminal status

### Services

- `Purchasing::PoLineQuantitySummary` — supply math (`open_supply_before_allocation_claims`)
- `Purchasing::SyncPoLineVendorQuantitiesFromSourcing` — aggregate/idempotent from final vendor responses
- `DemandAllocations::InboundAvailability` — claimable inbound (subtracts legacy + v0.04 claims)
- `DemandAllocations::ConvertInboundFromReceipt` — FIFO convert, split partials, idempotent
- `DemandAllocations::ReleaseUncoveredInbound` — reverse-FIFO release
- `Purchasing::ReceiptPostingGuards` — blocks mixed legacy + v0.04 inbound claims
- `Purchasing::InboundAvailabilitySnapshot` — read model for PO/demand surfaces
- `Purchasing::PostReceipt` — single transaction: inventory, PO update, legacy allocator, v0.04 conversion

### UI

- PO show: Requested / Accepted to stock / Open to receive; vendor bucket flags
- Receipt show/form: Expected / Counted / Accepted / Rejected labels
- Variant ops drawer: Open to receive

## Hard boundaries preserved

- Only `quantity_accepted` posts inventory
- v0.04-9 services do not write `purchase_order_line_allocations` / `receipt_line_allocations`
- Inbound→on_hand: new `on_hand` row + inbound `converted` (no `allocation_kind` mutation)
- `vendor_backorder` allocations excluded from receipt conversion
- Legacy `Receiving::AllocateCustomerDemandFromReceipt` unchanged
- `quantity_cascaded` on PO lines deferred
- POS `demand_allocation_id` deferred
- Posted receipt reversal deferred
