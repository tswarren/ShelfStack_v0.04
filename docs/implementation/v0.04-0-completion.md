## Data strategy

**Decision: reseed (not one-time migration script).**

### Rationale

* ShelfStack is **not in production**; destructive schema changes are acceptable per [delivery roadmap](../roadmap/v0.04-delivery-roadmap.md).
* Seeds are idempotent and validated via `rails shelfstack:seeds:validate`.
* v0.04-1+ will substantially reshape catalog/product tables; reseed after major milestones is simpler than incremental data migration during development.
* No production customer or transaction history requires preservation at this stage.

### Operational rule (v0.04-1 onward)

After destructive migrations during development:

```bash
./dev/rails-docker bin/rails db:drop db:create db:migrate db:seed
./dev/rails-docker bin/rails shelfstack:seeds:validate
```

### Exception policy

If a future milestone requires preserved demo or QA datasets, revisit migration tooling for that milestone only. Not required for v0.04-0 or initial v0.04-1 work.
