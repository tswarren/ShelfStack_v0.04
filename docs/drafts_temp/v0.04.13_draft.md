## Job

v0.04-13 makes ShelfStack’s demand, sourcing, ordering, and receiving workflows continuous for staff while ensuring those workflows can later be driven by vendor integrations such as EDI, API, file imports, vendor portals, or email-based processes.

This milestone does **not** implement X12, SFTP, EDI parsing, EDI generation, vendor APIs, automated polling, or invoice/AP processing.

It does make the core workflow ready for those future capabilities.

The target workflow is:

```
DemandLine
  → sourcing / vendor capability workflow
  → vendor response or order-to-confirm process
  → draft PO planned coverage
  → submitted/eligible PO inbound allocation
  → vendor shipment receiving workpad
  → receipt line matching
  → receipt posting
  → inventory posting
  → inbound allocation conversion
  → on-hand reservation / shelf stock
  → POS pickup or normal sale
```

---

## Purpose

v0.04-13 answers:

* How does ShelfStack know whether a vendor supports stock checks, order-to-confirm, manual review, EDI, API, portal, email, or file workflows?  
* How does demand move into a draft PO without pretending the draft PO is committed inbound supply?  
* How can staff see which demand a draft PO line is intended to cover?  
* How can a submitted or acknowledged PO become active inbound supply?  
* How can receiving start from a vendor shipment instead of a single PO?  
* How can one vendor shipment be matched across multiple open POs?  
* How can receiving prioritize customer demand, backorders, and shelf replenishment?  
* How do we make future vendor documents safe to import without changing the core domain model?  
* How do we prevent future EDI transport state from being confused with business state?

---

# 1\. Core Principles

## 1.1 Manual and integrated workflows use the same domain services

Future integrations must not become a parallel ordering system.

Manual staff actions and future vendor documents must call the same ShelfStack domain services:

```
DemandAllocations::AllocateInboundPurchaseOrder
Sourcing::RecordVendorResponse
Purchasing::SyncPoLineVendorQuantitiesFromSourcing
Purchasing::BuildPurchaseOrderFromDemand
Purchasing::AddDemandToPurchaseOrder
Purchasing::PostReceipt
DemandAllocations::ConvertInboundFromReceipt
Inventory::Post
```

Future EDI/API/file import services should translate external documents into ShelfStack domain actions.

They should not directly mutate inventory, demand, PO, or receipt state.

---

## 1.2 Document semantics must remain separate

ShelfStack must preserve these distinctions:

| Concept | Meaning | What it must not be treated as |
| :---- | :---- | :---- |
| Availability evidence | Vendor says stock may be available | Reservation, inbound supply, or inventory |
| Purchase order submission | Store asks vendor to supply goods | Vendor acceptance |
| Technical acknowledgment | Vendor/system confirms a file/message was syntactically received | Business acceptance |
| Vendor acknowledgment | Vendor confirms, backorders, cancels, or rejects requested quantity | Physical receipt |
| Shipment notice | Vendor says goods are shipping or shipped | Posted receipt |
| Invoice | Vendor bills for goods/services | Physical receipt or inventory event |
| Physical receipt | Store counts and accepts goods | Invoice approval |

---

## 1.3 Inventory posting remains unchanged

Inventory changes only through approved source-document posting.

For this milestone:

```
Accepted receipt quantity
  → Purchasing::PostReceipt
  → Inventory::Post
  → inventory_postings
  → inventory_ledger_entries
  → inventory_balances
```

Demand, sourcing, vendor availability, planned PO coverage, shipment notices, receipt matching, acknowledgments, and invoices must not directly post inventory.

---

## 1.4 Draft PO coverage is not inbound supply

Draft POs may show planned demand coverage.

They must not create active `inbound_purchase_order` demand allocations until the PO line is eligible under existing inbound availability rules.

```
Draft PO line
  → planned demand coverage only

Submitted/eligible PO line
  → may create inbound_purchase_order allocation
```

---

## 1.5 Shipment-first receiving

Receiving should start from what physically arrived or is expected to arrive from a vendor.

A receipt may involve:

```
one PO
multiple POs
no PO
a mix of PO-backed and unmatched lines
overages
shortages
substitutions
damaged items
wrong items
direct vendor shipment lines
```

The user should not be required to know which PO a vendor fulfilled before beginning receiving.

# 2\. Vendor Capability and Channel Model

## 2.1 Current issue

ShelfStack already models demand, sourcing attempts, vendor responses, POs, and receipts. However, vendor workflows differ significantly.

Examples:

```
Ingram:
  availability may later come from EDI 846
  orders may later be sent as EDI 850
  acknowledgments may later arrive as EDI 855
  shipment notices may later arrive as EDI 856
  invoices may later arrive as EDI 810

Small publisher:
  availability checked by email or phone
  orders sent by email
  acknowledgments recorded manually
  no formal shipment notice
  invoice arrives by email or paper

Local vendor:
  no stock check
  order to confirm
  partial shipment possible
  manual invoice
```

v0.04-13 should make these differences configurable without implementing the integrations.

---

## 2.2 Operational capability

Add or expose vendor capability fields.

```
availability_workflow:
  check_before_order
  order_to_confirm
  manual_review
```

### Meanings

| Value | Meaning |
| :---- | :---- |
| `check_before_order` | Staff or integration can check availability before order submission |
| `order_to_confirm` | Store must submit/order first; availability is learned afterward |
| `manual_review` | Buyer manually determines the next step |

# 3\. Future Vendor Document Mapping

v0.04-13 does not implement external documents, but it should define where they will map later.

## 3.1 Future document mapping

| Future vendor document | ShelfStack target |
| :---- | :---- |
| Inventory availability / future X12 846 | Vendor availability evidence / availability cache |
| Purchase order / future X12 850 | Outbound `purchase_order` submission |
| Purchase order acknowledgment / future X12 855 | `vendor_responses` \+ PO line vendor quantity buckets |
| Shipment notice / future X12 856 | Draft receiving workpad / expected shipment |
| Invoice / future X12 810 | Invoice/cost review, not inventory |
| Technical acknowledgment / future X12 997 or 999 | Transmission/control status only |

---

## 3.2 Hard semantic rules

```
Availability data is not reserved stock.
Purchase order submission is not vendor acceptance.
Technical acknowledgment is not business acknowledgment.
Vendor acknowledgment is not physical receipt.
Shipment notice is not posted receipt.
Invoice is not physical receipt.
Physical receipt is not invoice approval.
```

---

## 3.3 No EDI-specific state in core workflow names

Core workflow statuses should remain vendor-neutral.

Avoid statuses like:

```
edi_sent
edi_accepted
edi_rejected
x12_confirmed
```

Prefer domain states:

```
submitted
acknowledged
confirmed
partially_confirmed
backordered
canceled
closed_short
received
posted
```

EDI transport state should live in future integration records or external references, not overload PO, demand, or receipt status.

## 4.3 Rules

```

6. Future imported documents should use external references or idempotency keys to prevent duplicate processing.

---

## 4.4 Idempotency requirement

Any v0.04-13 service that may later be called by an integration must be idempotent or accept an idempotency key.

Integration-ready services include:

```
Sourcing::RecordVendorResponse
Purchasing::CreateDemandCoveragePlans
Purchasing::ConvertDemandCoveragePlansToInbound
Receiving::SuggestReceiptLineMatches
Receiving::ApplyReceiptLineMatches
Purchasing::PostReceipt
```

Rules:

```
Same external reference + same domain target
  → must not create duplicate vendor responses, demand plans, allocations, receipt lines, or matches.

Retry after timeout
  → must be safe.

Partial failure
  → must roll back or leave a recoverable processing state.
```

# 5\. Vendor Availability Evidence

## 5.1 Purpose

Vendor availability evidence helps buyers select vendors and decide whether demand should be ordered, backordered, cascaded, or manually reviewed.

It is not committed supply.

---

## 5.2 Rule

Vendor availability data may suggest supply.
It must not reserve vendor stock.
It must not create inbound purchase order allocations.
It must not mark demand allocated.
It must not post inventory.

```

```

Outputs:

```

```

---

## 6.2 Staff-facing action labels

| Vendor setup | Label | Meaning |
| :---- | :---- | :---- |
| `availability_workflow = check_before_order` | Check availability | Check before committing to order |
| `availability_workflow = order_to_confirm` | Order to confirm | Submit/add to PO before availability is known |
| `availability_workflow = manual_review` | Record manual response | Buyer records vendor result manually |
| existing backorder | Accept backorder | Demand remains allocated to vendor backorder |
| failed/unavailable | Try next vendor | Buyer-reviewed cascade |
| no vendor | Choose vendor | Buyer selects source |

---

## 6.3 Order-to-confirm behavior

For vendors without pre-order verification:

```
DemandLine unresolved quantity
  → SourcingRun
  → SourcingAttempt
  → Draft PO planned coverage
  → PO submitted / order sent
  → Vendor response recorded later
  → PO vendor quantity buckets updated
  → demand becomes inbound, vendor_backorder, unresolved, or canceled
```

Key rule:

```
A PO line for an order-to-confirm vendor is requested supply, not confirmed supply.
```

---

## 6.4 Check-before-order behavior

For vendors with availability checks:

```
DemandLine unresolved quantity
  → SourcingRun
  → SourcingAttempt
  → availability evidence / vendor response
  → confirmed quantity may be added to PO
  → backordered quantity may become vendor_backorder
  → unresolved quantity remains for cascade or review
```

---

## 6.5 Manual review behavior

For manual vendors:

```
DemandLine unresolved quantity
  → SourcingRun
  → SourcingAttempt
  → buyer records manual response
  → confirmed/backordered/unavailable/substitute/canceled handled normally
```

# 7\. Durable Draft PO Demand Coverage

## 7.1 Current issue

v0.04-12 can build POs from demand, but draft PO demand context is audit-level rather than durable line-level data.

v0.04-13 adds planned demand coverage so staff can see why each draft PO line exists.

---

## 7.2 Design principle

```
Draft PO coverage = planned coverage.
Submitted/eligible PO coverage = inbound allocation.
```

Planned coverage is visible and durable, but does not affect inventory or availability.

---

## 7.3 New table: `purchase_order_line_demand_plans`

### Grain

One row represents:

```
A planned quantity from one demand line intended to be covered by one purchase order line.
```

### Suggested fields

```
id
store_id
purchase_order_id
purchase_order_line_id
demand_line_id
product_id
product_variant_id
quantity_planned
coverage_kind
status
created_by_user_id
converted_to_demand_allocation_id
converted_at
converted_by_user_id
released_at
released_by_user_id
release_reason
idempotency_key
notes
created_at
updated_at
```

### `coverage_kind` values

```
customer_fulfillment
shelf_replenishment
frontlist_stock
display_stock
event_stock
preorder_fulfillment
backorder_fulfillment
replacement
other
```

### `status` values

```
planned
partially_converted
converted
released
canceled
superseded
```

---

## 7.4 Rules

1. Planned coverage does not affect `quantity_available`.  
2. Planned coverage does not affect `quantity_reserved`.  
3. Planned coverage does not create inventory ledger entries.  
4. Planned coverage does not create active inbound allocations while the PO is draft.  
5. Planned coverage must match store, product, and variant across demand line and PO line.  
6. Planned quantity must be positive.  
7. Planned coverage should be idempotent by demand line \+ PO line \+ source action.  
8. Planned coverage should survive draft PO edits until released, converted, canceled, or superseded.

---

## 7.5 Draft PO creation from demand

When creating a draft PO from demand:

```
DemandCoveragePlanner
  → BuildPurchaseOrder
  → purchase_order_lines
  → purchase_order_line_demand_plans
```

The audit event remains useful, but planned coverage records become the durable source of truth for line-level demand intent.

---

## 7.6 Add demand to existing draft PO

When adding demand to an existing draft PO:

1. Confirm PO is draft.  
2. Resolve demand against the PO vendor.  
3. Add or update PO lines.  
4. Create planned coverage rows.  
5. Record audit event.  
6. Refresh PO demand breakdown.

---

## 7.7 Editing draft PO quantity

If a draft PO line quantity is reduced below active planned coverage quantity, staff must reconcile coverage.

Options:

```
release planned coverage
reduce shelf replenishment coverage
reduce newest planned coverage
manually choose coverage to release
cancel edit
```

Customer fulfillment coverage should require explicit confirmation before release.

---

## 7.8 Conversion to inbound allocation

When a PO line becomes eligible inbound supply:

```
purchase_order_line_demand_plans
  → DemandAllocations::AllocateInboundPurchaseOrder
  → demand_allocations allocation_kind = inbound_purchase_order
```

Conversion rules:

1. Use existing inbound availability rules.  
2. Convert active planned coverage up to available inbound quantity.  
3. Create active `inbound_purchase_order` demand allocations.  
4. Link plan row to created allocation.  
5. Mark plan row converted or partially converted.  
6. Recalculate demand status.  
7. Audit conversion.  
8. Do not duplicate conversion on retry.

# 8\. Purchase Order Submission Readiness

## 8.1 PO business status vs submission status

PO business status should not be overloaded with future EDI transport state.

Purchase order status answers:

```
Where is this PO in the purchasing business lifecycle?
```

Transmission state answers:

```
How, whether, and with what result was this PO sent to the vendor?
```

In v0.04-13, transmission state may remain audit/external-reference based.

Future integration records may track:

```
not_submitted
ready_to_submit
queued
transmitted
technical_acknowledged
technical_rejected
business_acknowledged
business_rejected
```

But those should not replace PO business statuses.

---

## 8.2 Submission method

Add or expose a submission method concept:

```
submission_method:
  manual
  email
  portal
  file_export
  edi_x12
  api
```

This may be stored as an external reference, audit detail, or future integration record.

Minimum v0.04-13 behavior:

* manual submission remains supported;  
* vendor configuration indicates preferred order submission method;  
* future automated submission can attach external references without changing PO lifecycle.

# 9\. Vendor-Shipment-First Receiving

## 9.1 Current issue

Vendors may combine items from multiple POs into a single shipment.

A PO-first receiving workflow is too narrow.

---

## 9.2 Design principle

```
Receiving starts from the vendor shipment.
PO matching happens underneath.
```

A receiving workpad may be created manually now and by a future shipment notice later.

---

## 9.3 Shipment notice readiness

Add this rule:

```
A shipment notice is not a physical receipt.

A future imported shipment notice may prefill a draft receiving workpad.
It must not post inventory.
It must not mark demand fulfilled.
It must not prove that goods physically arrived.
```

Physical receiving remains the point where staff records:

```
quantity_expected
quantity_received
quantity_accepted
quantity_rejected
exception reason
```

Only accepted quantity posts inventory.

---

# 10\. Receipt Line Matching

## 10.1 Matching candidates

When staff enters a receipt line, ShelfStack should suggest matching PO lines.

Candidate criteria:

```
same store
same vendor
same product_variant
PO not canceled/closed
PO line not closed/canceled/fully received
open_to_receive > 0
```

Suggested sort priority:

1. Customer fulfillment demand.  
2. Customer-paid or urgent demand.  
3. Preorder/backorder fulfillment.  
4. Older customer demand.  
5. Event/display demand.  
6. Older submitted PO lines.  
7. Confirmed vendor quantity.  
8. Shelf replenishment.  
9. General stock.

---

## 10.2 Matching approach

v0.04-13 may choose one of two approaches.

### Option A — auto-split receipt lines

One scanned shipment line can become multiple receipt lines internally.

Example:

```
Shipment line:
  Book A, quantity 5

Internal receipt lines:
  qty 1 → PO #100 line
  qty 1 → PO #104 line
  qty 3 → PO #109 line
```

Pros:

```
smaller schema change
uses existing receipt_lines.purchase_order_line_id
simpler posting integration
```

Cons:

```
repeated lines for staff
physical shipment line not preserved cleanly
less flexible for future shipment notice import
```

---

### Option B — new `receipt_line_matches`

Add a v0.04-native matching table.

This is not the retired v0.03 `receipt_line_allocations` table.

It is a shipment-to-PO matching table.

Suggested fields:

```
id
store_id
receipt_id
receipt_line_id
purchase_order_id
purchase_order_line_id
product_id
product_variant_id
quantity_matched
match_status
match_source
matched_by_user_id
matched_at
released_by_user_id
released_at
release_reason
idempotency_key
notes
created_at
updated_at
```

### `match_status` values

```
proposed
confirmed
posted
released
rejected
```

### `match_source` values

```
auto
manual
override
file_import
edi_x12
api
```

Pros:

```
preserves physical shipment line
supports one line matched to multiple PO lines
future shipment notices map cleanly
better audit
better receiving UI
```

Cons:

```
larger implementation
requires post integration
requires verifier coverage
```

---

## 10.3 Recommendation

Prefer **Option B** if v0.04-13 is intended to make receiving truly integration-ready.

Use Option A only if the milestone must be kept smaller.

---

## 10.4 Rule for receipt line matches

```
Receipt line matches are shipment-to-PO matching records.

They are not demand allocations.
They do not reserve stock.
They do not post inventory.
They do not fulfill demand.

They only explain how received or expected shipment quantity maps to PO lines.
```

```

---

## 11.4 Unmatched quantity

If quantity cannot be matched to a PO line, staff may:

```
receive as direct/unplanned stock
search broader PO history
mark as substitution review
mark as wrong item
mark as damaged
create/find product and variant
flag for possible RTV
leave for buyer review
```

Unmatched accepted quantity may post inventory only after resolving to an inventory-eligible product variant.

---

# 12\. Receipt Posting

## 12.1 Posting authority

`Purchasing::PostReceipt` remains the only receipt posting authority.

---

## 12.2 Atomic transaction

Posting must occur in one transaction:

```
validate receipt draft
validate line quantities
validate inventory eligibility for accepted lines
record discrepancies
confirm receipt line matches / split lines
post accepted quantity through Inventory::Post
update PO line received/accepted quantities
convert inbound demand allocations to on-hand
update planned coverage where applicable
rebuild availability cache
recalculate demand lines
mark receipt posted
record audit events
```

If any critical step fails, the entire post rolls back.

---

## 12.3 Demand allocation conversion

Existing v0.04-9 behavior remains:

```
accepted PO-backed receipt quantity
  → convert active inbound_purchase_order allocations FIFO
  → create new active on_hand allocations
  → terminalize inbound rows as converted
  → rebuild availability cache
  → recalculate demand status
```

v0.04-13 should improve preview and traceability around this behavior.

---

## 12.4 Short, rejected, and closed-short behavior

Keep existing rules:

* Rejected quantity does not automatically release demand.  
* Demand releases only when effective inbound supply is reduced or staff explicitly releases it.  
* Closed-short quantity may release affected inbound allocations.  
* Vendor-canceled quantity may release affected inbound allocations.  
* Rejected-not-replaceable may release affected demand when staff explicitly chooses that action.

---

# 13\. Invoice and Cost Boundary

## 13.1 Rule

Invoices are cost/billing evidence, not receiving evidence.

A future invoice import may:

```
create invoice/cost review records
flag cost mismatches
compare invoice cost to PO cost
compare invoice cost to receipt cost
support future AP workflows
```

It must not:

```
post inventory
replace physical receiving
mark demand fulfilled
convert inbound allocations to on-hand
```

# 23\. Milestone Outcome

After v0.04-13, ShelfStack should support a clean manual-first workflow:

```
Demand is captured.
Buyer sees unresolved demand.
Vendor capability determines the next action.
Demand can populate draft PO planned coverage.
Submitted/eligible PO lines convert planned coverage to inbound allocations.
Vendor responses update PO supply state.
Receiving starts from the vendor shipment.
Shipment lines are matched across one or more POs.
Accepted stock posts inventory.
Inbound demand converts to on-hand reservations.
Customer demand becomes ready for pickup.
Shelf demand becomes available stock.
Unresolved demand remains in sourcing, backorder, release, or review.
```

It should also be ready for future integrated workflows:

```
Vendor availability import
  → availability evidence

Outbound order document
  → purchase order submission trace

Vendor acknowledgment
  → vendor_response + PO quantity buckets

Shipment notice
  → draft receiving workpad

Invoice
  → cost/invoice review

Technical acknowledgment
  → transmission status only
```

The core rule remains:

```
Vendor integrations may provide evidence, documents, references, and workflow triggers.

They must not bypass ShelfStack’s demand, sourcing, purchasing, receiving, and inventory posting services.
```

This version should be safer as a foundation for future X12 without pulling EDI implementation into v0.04-13.
