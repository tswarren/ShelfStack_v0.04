# v0.04-14 — Product/Catalog Cleanup Wireframes

## Status

**Companion planning artifact** for [v0.04-14 product/catalog cleanup](v0.04-14-product-catalog-cleanup.md).

These are low-fidelity text wireframes. They are intentionally concrete about page organization, information hierarchy, labels, and primary actions, but they do not prescribe final CSS or component implementation.

## How to read these wireframes

These wireframes are **target-state sketches for v0.04-14**, not screenshots of the current v0.04-13 app. Current screens may still show pre-cleanup labels such as CatalogItem, `catalog_item_id`, legacy catalog identifiers, Manual TBO, or retained import/admin paths. The wireframes show where those concepts should land **after** the cleanup: canonical work stays product/variant-native, while catalog/import details move to secondary legacy/provenance regions.

Each screen includes required behavior for implementation review. Treat those bullets as acceptance criteria for the future cleanup, not claims that current screens already behave that way.

## Layout principles

1. **Product is the item.** Every ordinary item screen starts from a product record, not a catalog item record.
2. **Variant is the operational unit.** Inventory, POS, demand, ordering, receiving, buyback, and reservation facts appear at variant grain.
3. **Catalog/import is provenance.** Legacy catalog and provider payload details are secondary context, not primary navigation.
4. **One primary action per surface.** Secondary actions belong in action groups, drawers, menus, or tab-local controls.
5. **Summary before tables.** Staff should see readiness, demand, inventory, and ordering signals before detailed rows.
6. **Bookstore density, not admin sprawl.** Pages may be dense, but each region needs a clear job.

## 1. Items index / product search

### Job

Help staff find an item or start an add-item flow without exposing legacy catalog objects as normal search results.

### Wireframe

```text

│ Filters: [All] [Books] [Sidelines] [Inventory warnings] [Open demand]       │
├──────────────────────────────────────────────────────────────────────────────┤
│ Summary                                                                      │
│ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────────────┐ │

│ Results                                                                      │
│ ┌──────────────────────────────────────────────────────────────────────────┐ │
│ │ Cover  Product / primary identifier        Variants  Stock  Demand  CTA │ │
│ │ [img]  The Left Hand of Darkness            3         4      2       Open│ │
```

## 8. Mobile/narrow fallback

### Job

Preserve hierarchy on narrow screens without losing the product-native mental model.

### Wireframe

```text
Product title
Primary identifier
[Primary action]

Summary cards
- Variants
- On hand
- Demand
- On order

Tabs become stacked sections:
1. Overview
2. Variants
3. Demand
4. Ordering
5. Inventory
6. Metadata
7. Setup
8. Activity

Legacy / import context remains collapsed at the bottom.
```

### Required behavior

* Header identity and primary action stay above operational summary.
* Legacy/import context stays below canonical product/variant content.
* Do not move legacy/admin actions into the mobile primary action slot.

## Review checklist

Use this checklist before implementation:

1. Can a staff user tell the page is about a Product within three seconds?
2. Is the first operational table or drawer at ProductVariant grain?
3. Is there exactly one obvious primary action in the current region?
4. Are legacy catalog/source details collapsed or clearly secondary?
5. Does external lookup resolve local Product/ProductIdentifier matches before create actions?
6. Does the product form avoid a routine `catalog_item_id` picker?
7. Do labels use product/demand-native terms rather than v0.03 catalog/request terms?
8. Does every compatibility exception have an owner and removal condition?
