# ShelfStack v0.04 — Delivery Roadmap

## Purpose

**ShelfStack v0.04 is the core domain model** for the application going forward — not another numbered phase in the v0.03 delivery history.

The v0.03 **Phases 1–10** built a working codebase on an earlier catalog and ordering vocabulary. v0.04 replaces that vocabulary with the canonical architecture for items, identifiers, demand, sourcing, and receiving. Implementation happens in ordered **milestones** (`v0.04-0` … `v0.04-11`) inside the existing Rails app; this is not a greenfield rewrite.

It answers:

> How should ShelfStack represent commercial items, sellable variants, identifiers, customer/store demand, vendor sourcing, and receiving so bookstore workflows operate at the correct grain?

**Core domain reference:** [VERSION_0.04.md](../design/VERSION_0.04.md)

**Strategy:** Preserve variant-grain operations (inventory, POS, buybacks, classification, foundation). Rebuild the item identity layer and the demand-to-fulfillment pipeline. Because ShelfStack is not yet in production, prefer **destructive schema changes** over long-lived compatibility shims.

**Document role:** This is the **roadmap and control document** for v0.04 — direction, invariants, lifecycle vocabulary, and milestone boundaries. Per-milestone specs, data models, and test plans live under [../v0.04/](../v0.04/) when scoped. Do not treat this file alone as an implementation spec.

---

## Relationship to v0.03

Phases 1–10 delivered the **v0.03 implementation**:

```text
Catalog Item → Product → Product Variant → Inventory / POS / Purchasing
```

v0.04 **is the new core** for the top of that stack and for ordering workflows:

| Carry forward from v0.03 | Replace under v0.04 core |
| ------------------------ | ------------------------ |
| Product variant as operational unit | `catalog_items` vs `products` separation |
| `Inventory::Post` / ledger / balances | `customer_requests`, `special_orders`, `purchase_requests` / TBO |
| POS transactions, voids, tax/discount/tender | PO/receipt allocation model as primary demand linkage |
| Purchase orders and receipts (document concepts) | Supplier availability assumptions |
| Buybacks, stored value, classification/tax | `catalog_item_identifiers` → `product_identifiers` |
| Foundation (auth, sessions, audit, workstations) | Overlapping reservation + allocation + request line concepts |
| Phase 10 interaction infrastructure (modals, drawers, POS workspace) | — |

Treat v0.03 phase specs as **historical reference** for code not yet migrated. Do not extend v0.03 catalog-item or customer-request patterns for new work.

Phase 10-E (app-wide consistency sweep) is **paused** until the v0.04 core stabilizes underneath operational screens.

## Target architecture

```text
Optional product_group          # never operational; see naming below
  → Product                    # specific commercial item (ISBN/UPC edition, release, sideline)
    → Product Variant          # sellable / stockable / reservable / orderable form
      → Demand               # customer or store need (does not mutate inventory)
        → Allocation         # claim on on-hand or inbound supply
          → Sourcing         # vendor attempts and responses
            → Purchase Order / Receiving
              → Inventory::Post
                → Fulfillment (POS pickup, ship, hold release)
```

**Identifiers (design doc §3):**

```text
validation_family: gtin | isbn | freeform | house
ISBN-10 ↔ ISBN-13 automatic alternates; gtin primary for books
House EAN-13: prefix 200–229, sequential allocator, EAN-13 check digit (product identifiers)
Variant SKU: system-assigned at variant creation; format TBD (sequential internal vs EAN-13)
```

## Core v0.04 invariants

These rules apply across all milestones. Per-milestone specs must not contradict them without an explicit design-doc update.

* A **product** represents one specific commercial item (one ISBN/UPC edition, release, or manufactured item) — not a family of editions.
* A **product variant** is the operational grain for POS, inventory, purchasing, receiving, reservations, allocations, and buybacks.
* A **product group** is optional, non-operational grouping only (navigation, search, related editions) — never POS, inventory, or PO grain.
* **Demand** records a need; it does **not** change on-hand inventory.
* **Allocation** claims or points demand toward supply (on-hand or inbound); it does **not** post inventory.
* Only **`Inventory::Post`** may mutate authoritative on-hand quantity.
* Only **accepted receipt quantity** posts to inventory (rejected/damaged/wrong/substituted/short does not).
* **Used variants** may be customer-reservable but are **not vendor-orderable**; unavailable used demand does not auto-convert to new-item PO lines.
* **Product identifiers** belong to products; **variant SKUs** are system-assigned operational codes for sellable forms — not derived from product identifiers.
* **Vendor availability** is not assumed at order time; confirmed quantity comes from **vendor response** (milestone v0.04-8 onward).

## Demand taxonomy (replacing “special order”)

Bookseller workflows map to v0.04 concepts as follows. Milestone **v0.04-6** implements this vocabulary; **v0.04-7** adds allocations.

| Bookseller action | v0.04 concept |
| ----------------- | ------------- |
| Reserve a copy already on hand | `demand_line` + `on_hand` allocation |
| Reserve the next copy already on order | `demand_line` + `inbound_purchase_order` allocation |
| Ask buyer to order a copy | `demand_line` unresolved → sourcing (v0.04-8) |
| Wait for a used copy | `demand_line` with `used_wanted` purpose/source; no vendor PO |
| Buyer note to consider stocking | `stock_consideration` — not committed demand |
| Manual TBO / store replenishment | `demand_line` with store source (e.g. `manual_tbo`, `buyer_decision`) |

There is no separate `special_orders` header in v0.04. Customer-facing “special order” language in UI may remain as a **label** for certain demand purpose/source combinations.

## Lifecycle vocabulary (state machines)

Authoritative transition rules belong in per-milestone specs. This section fixes **expected status names** so implementations do not diverge.

### `demand_lines`

```text
draft / captured
  → open
  → partially_allocated
  → allocated
  → fulfilled
  → canceled
  → expired
```

### `demand_allocations`

```text
active
  → fulfilled
  → released
  → expired
  → canceled
```

### `sourcing_attempts`

```text
pending
  → submitted
  → confirmed
  → partially_confirmed
  → backordered
  → canceled
  → failed
  → cascaded
```

### `purchase_order_lines` (quantity + header status)

Line-level quantity fields (requested, confirmed, backordered, etc.) are authoritative in v0.04-9. Header/line status vocabulary:

```text
draft
  → submitted
  → confirmed / partially_confirmed / backordered
  → partially_received
  → received
  → closed_short
  → canceled
```

### `receipt_lines`

```text
expected
  → received
  → accepted / rejected / damaged / wrong_item / substituted / short
  → posted
```

Only **accepted** quantity on posted receipt lines affects inventory and MAC.

## Vendor cascade (v0.04-8 policy)

Unresolved demand after a vendor response may move to the next sourcing attempt. **v0.04 baseline policy:**

* **Rule-driven vendor priority** — try configured vendor order (variant → product → defaults) before ad hoc overrides.
* **Partial confirmation splits quantity** — confirmed → inbound allocation; backordered → backorder allocation; canceled/unavailable → eligible for cascade.
* **Buyer review required for cascade** in v0.04 — automatic re-submission to the next vendor is **not** the default; buyer (or explicit staff action) approves cascade unless a later spec adds auto-cascade rules per vendor profile.

Example:

```text
Customer demand: 3 copies, new hardcover variant.

Sourcing attempt 1 — preferred wholesaler:
  quantity_requested: 3
  quantity_confirmed: 1
  quantity_backordered: 1
  quantity_canceled: 1

Result:
  → 1 inbound PO allocation (confirmed)
  → 1 vendor_backorder allocation (if store accepts backorder wait)
  → 1 quantity returns to unresolved / buyer review for cascade
  → Buyer may attempt alternate wholesaler, publisher direct, or cancel demand
```

Wholesaler vs publisher-direct vs manufacturer-direct attempts differ by discount, minimums, returnability, and lead time — capture on vendor sourcing records (below).

## Sourcing hierarchy

Preserve and extend existing v0.03 joins. Resolution order for suggested vendor and vendor item number:

```text
product_variant_vendors
  → product_vendors
  → vendor / store defaults
  → manual buyer override (audited)
```

Each sourcing record should eventually support (v0.04-8 spec detail):

| Field / concern | Notes |
| --------------- | ----- |
| Vendor item number | Ordering identifier at vendor |
| Vendor priority | Cascade order hint |
| Default discount / net cost | Snapshotted on PO submit (existing Phase 5 behavior) |
| Returnability | Existing precedence chain preserved |
| Minimum order quantity | Optional; blocks or warns at submit |
| Availability method | `manual`, import, API, EDI/Pubnet — **later** |
| Last confirmed availability | From vendor response, not guess at order time |
| Lead time estimate | Operational hint only in v0.04 |
| active | Prefer inactivation over delete |

## Invoice and cost reconciliation boundary

**Deferred outside v0.04 core:** full accounts-payable invoice matching, freight allocation, and GL-shaped cost variance (Phase 9c territory).

**In scope for v0.04-9 receiving:** receipt lines must snapshot enough cost data for inventory MAC and later invoice review:

* Ordered/expected unit cost (from PO line snapshot)
* Received unit cost (actual on receipt if different)
* Accepted quantity only updates moving average
* Optional `invoice_cost_cents` / variance reason fields may be added on receipt lines if cheap — not required for first v0.04-9 slice

Do not block v0.04 receiving on invoice workflow completeness.

## Product groups — canonical naming

**Table/model name:** `product_groups` (not `works`).

**Display:** “Work” is a **group type** for bibliographic grouping, not the universal label for all grouped merchandise.

```text
product_groups
  group_type: work | series_cluster | release_family | merchandise_family
```

Examples:

* Book editions sharing a title → `group_type: work`
* Sideline colorways → `group_type: merchandise_family`
* Box set / multi-format cluster → `group_type: release_family`

Product groups remain non-operational (see invariants).

## v0.04-1 — Product Model Fusion

### Job

Collapse `catalog_items` into `products` so a product is the specific commercial item (hardcover ISBN, paperback ISBN, CD UPC, gift UPC).

### Major changes

* Move catalog metadata fields onto `products` (title, subtitle, format, publisher/creator JSONB, series, subjects, thumbnails, etc.).
* Remove `products.catalog_item_id` and the `catalog_items` table.
* Enforce **one product per commercial item** (not one product spanning hardcover + paperback).
* Update `Product`, seeds, and factory/fixture patterns.
* Retire `CatalogItem` model and catalog-item-specific controllers where superseded by product routes.

### Services and UI to update (initial pass)

* Add Item wizard (`AddItem::*`)
* External catalog lookup and import (`ExternalCatalog::*`, `IngramCatalogImport::*`)
* `Items::ItemPresenter`, item overview and index
* Product name rendering (`ProductNameRenderer`)

### Preserve

* `product_variants` table and operational fields
* Classification defaults resolver chain (variant → product → format/type)
* Vendor sourcing joins at product / variant level

### Definition of done

* No remaining runtime dependency on `catalog_items` for create/read/update flows.
* Seeds and Add Item create **products** with metadata directly.
* Tests for item creation and external import pass on new model.

## v0.04-2 — Product Identifiers

### Job

Replace `catalog_item_identifiers` with `product_identifiers` and validation families.

### Major changes

* New `product_identifiers` table with `validation_family`: `gtin`, `isbn`, `freeform`, `house`.
* New `ProductIdentifierService` (successor to `CatalogIdentifierService`):
  * GTIN check-digit validation for `gtin` and `house`
  * Mod-11 validation for `isbn`
  * **ISBN-10 entry** → auto-create primary `gtin` ISBN-13 (existing behavior)
  * **ISBN-13 entry (`978…`)** → auto-create non-primary `isbn` ISBN-10 (new)
  * **No ISBN-10 alternate for `979…`**
* House EAN-13 generation: prefix **200–229**, **sequential** counter per segment, **EAN-13** check digit (not UPC-A weighting) — for **product identifiers**, not variant SKUs.
* Global uniqueness on normalized GTIN values; separate uniqueness scope for `isbn` and `freeform`.
* Drop `catalog_item_identifiers`.
* **Variant SKU allocator (open decision):** replace v0.03 identifier-derived `SkuGenerator` with system-assigned variant SKUs — either sequential internal codes or dedicated EAN-13 segment; choose in v0.04-2 data model.

### Preserve

* POS ambiguity when multiple variants match one product identifier scan
* `lookup_candidates` cross-form behavior for product identifiers
* Variant SKU **uniqueness** and **no reuse after inactivation** (v0.03 rules)

### Replace (v0.03 behavior)

* Do **not** derive variant SKU from product identifier, `products.sku`, or condition/attribute suffixes.
* `SkuGenerator` assigns variant SKU at creation; product identifier service does not supply variant barcode values.

### Definition of done

* Identifier CRUD, validation previews, and audit events work on products.
* Bidirectional ISBN alternates covered by service tests.
* House allocator: sequential EAN-13 in prefix 200–229 with segment policy; EAN-13 (not UPC) check digit — product identifiers only.
* Variant SKU format decided and documented in v0.04-2 data model (sequential internal **or** system EAN-13).
* `SkuGenerator` assigns system variant SKUs; does not read product identifiers for barcode values.

## v0.04-5 — Used Variant Rules

### Job

Codify used-product behavior on variants and align buyback/demand hooks.

### Major changes

* Explicit variant flags / enums: `vendor_orderable`, `customer_reservable`, `replenishment_strategy` (e.g. `vendor`, `used_acquisition_only`).

## v0.04-6 — Demand Foundation

### Job

Introduce unified **demand lines** and **stock considerations**; begin retiring fragmented request concepts.

### Major changes

* `demand_lines` with source, purpose, status, quantity, variant FK, customer FK (optional), due context — use [Lifecycle vocabulary](#lifecycle-vocabulary-state-machines) and [Demand taxonomy](#demand-taxonomy-replacing-special-order).
* `stock_considerations` for non-committing buyer notes (open → converted/dismissed).
* Demand sources include `customer_order`, `manual_tbo`, `used_wanted`, `buyer_decision`, etc. (see design doc §7).
* Services: create, cancel, transition status, match variant from provisional capture.
* **Parallel operation:** v0.03 customer request tables may remain read-only until v0.04-10; avoid dual-write long term.

### Replaces (eventually)

* `customer_request_lines` as the primary demand record
* `special_orders` as a separate header concept (demand carries purpose instead)
* `purchase_request_lines` / TBO as separate demand type (demand source = `manual_tbo`, etc.)

### Definition of done

* Staff can create demand from item variant, customer workspace, and manual entry.
* Each row in [Demand taxonomy](#demand-taxonomy-replacing-special-order) has a representable path (or explicit deferral documented in milestone spec).
* Stock considerations convert to demand or dismiss without inventory effect.
* Used wanted demand does not enter vendor sourcing queue.

## v0.04-7 — Allocations and Reservations

### Job

Unify **demand allocations** and **inventory reservations** under one conceptual model.

### Major changes

* `demand_allocations` with types: `on_hand`, `inbound_purchase_order`, `draft_receipt`, `vendor_backorder`, `waitlist`, `preorder`, `used_wanted`.
* Bridge or replace `inventory_reservations`, `purchase_order_line_allocations`, `receipt_line_allocations`.
* Receiving and PO screens show demand context via allocation joins, not ad hoc FK chains.
* POS pickup lines reference allocation/reservation consistently.
* `quantity_available` continues to respect active on-hand holds.

### Preserve

* Reservations do not post inventory
* FIFO receipt allocation behavior (unless explicitly revised in v0.04-9 spec)
* Override and expiry rules from v0.03 customer demand (Phase 7A)

### Definition of done

* Customer fulfillment path traceable: demand → allocation → PO/receipt/stock → pickup.
* Partial receipt updates allocations atomically with `PostReceipt`.

## v0.04-9 — PO and Receiving Quantity Model

### Preserve

### Definition of done

## Cross-cutting rules (all milestones)

1. **Controllers stay thin** — business rules in services (`ProductIdentifierService`, demand/allocation/sourcing services).
2. **Authorization and audit** — preserve foundation patterns; new mutations get permission keys and audit events.
3. **Tests required** — especially identifier validation, demand transitions, allocation integrity, and POS lookup.
4. **Seeds idempotent** — stable keys; validate with `rails shelfstack:seeds:validate`.
5. **No silent domain drift** — if implementation disagrees with [VERSION_0.04.md](../design/VERSION_0.04.md), update the design doc or flag the conflict.
