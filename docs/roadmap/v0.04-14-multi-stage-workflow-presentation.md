# v0.04-14 — Multi-Stage Workflow Presentation

## Status

**Companion target-state planning artifact** for [v0.04-14 product/catalog cleanup](v0.04-14-product-catalog-cleanup.md) and [wireframes](v0.04-14-product-catalog-wireframes.md).

This document defines how product/catalog cleanup workflows should present multi-stage work without reverting to pre-v0.04 CatalogItem-first or request/TBO-first mental models.

## How to read this document

These patterns describe the **target presentation model** for implementation. Current v0.04-13 screens may already support parts of these flows, may use different names, or may expose retained compatibility paths. The goal is to make future cleanup slices use one consistent workflow shell.

## Workflow presentation principles

1. **Show the canonical record first.** The header names the Product, ProductVariant, DemandLine, PO, or Receipt that owns the workflow.
2. **Show progress, not implementation tables.** Staff should see the stage they are in, what has already happened, and what remains.
3. **Keep the work surface local.** Use the current page, modal, drawer, or workpad; avoid sending staff through unrelated admin pages.
4. **Keep exceptions close to the action.** Warnings and blockers appear beside the stage/action that can resolve them.
5. **Summarize before line detail.** Each stage starts with a short summary and then reveals rows, matches, payloads, or history.
6. **Make legacy/provenance context secondary.** Import payloads, legacy CatalogItem links, and repair tools are secondary panels, not workflow drivers.
7. **Support save-and-return.** Multi-stage work should tolerate interruption and make the next action obvious on return.

## Standard workflow shell

Use this structure for any v0.04-14 multi-step workflow:

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Workflow title                                             [Save] [Exit]    │
│ Canonical context line: Product / Variant / Demand / PO / Receipt            │
├──────────────────────────────────────────────────────────────────────────────┤
│ Progress                                                                     │
│ [1 Identify] → [2 Review] → [3 Configure] → [4 Confirm] → [5 Complete]      │
├──────────────────────────────────────────────────────────────────────────────┤
│ Stage summary                                                                │
│ What is true now? What needs attention? What happens if the user continues? │
├──────────────────────────────────────────────────────────────────────────────┤
│ Main work surface                                                            │
│ Stage-specific form, match table, line editor, or preview.                  │
├──────────────────────────────────────────────────────────────────────────────┤
│ Supporting context                                                           │
│ Collapsed provenance, legacy/admin, history, and audit detail.              │
├──────────────────────────────────────────────────────────────────────────────┤
│ Footer actions                                                               │
│ [Back]                                            [Continue / Complete]     │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Required regions

| Region | Required content |
| --- | --- |
| Header | Workflow name, canonical owner, exit/save controls. |
| Progress | Step labels in staff language, not table/service names. |
| Stage summary | Current facts, readiness, blockers, and next effect. |
| Main work surface | The only area where the current step is edited. |
| Supporting context | Collapsed source, history, raw payload, or legacy repair detail. |
| Footer actions | Back/continue/complete actions with explicit disabled/blocker states. |

## Stage language rules

Use verbs staff understand:

```text
Identify → Review → Configure → Confirm → Complete
Find → Match → Choose → Preview → Apply
Intake → Price → Decide → Pay → Complete
Select demand → Build PO → Review impact → Submit
Receive shipment → Match lines → Preview impact → Post receipt
```

Avoid stage names that expose implementation internals:

```text
Catalog item create
Purchase request conversion
TBO import
Allocation mutation
Adapter output
Raw payload apply
```

## Pattern A — Identify / Review / Apply

Use for add-item, external lookup, import, and metadata fill-blank workflows.

```text
Stage 1 — Identify
  Input: ISBN / UPC / title / author / SKU
  Output: local matches + external candidates

Stage 2 — Review
  Input: selected local product or external candidate
  Output: field comparison and recommended action

Stage 3 — Apply
  Input: create product, fill blanks, link source, or no action
  Output: product detail / identifier review / variant setup
```

### Target wireframe

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Add Item from Lookup                                      [Save draft] [Exit]│
│ Canonical destination: Product                                               │
├──────────────────────────────────────────────────────────────────────────────┤
│ [1 Identify] → [2 Review candidate] → [3 Apply to product]                  │
├──────────────────────────────────────────────────────────────────────────────┤
│ Stage summary                                                                │
│ Found one local Product match and one provider candidate.                    │
│ Recommended: fill blank metadata on existing Product.                        │
├──────────────────────────────────────────────────────────────────────────────┤
│ Candidate comparison                                                         │
│ Field              Current Product              Provider candidate           │
│ Title              The Left Hand of Darkness     The Left Hand of Darkness   │
│ Publisher          —                            Ace                          │
│ Publication date   —                            1976-07-01                   │
│                                                                              │
│ Action                                                                           
│ ● Fill blanks on existing Product                                             │
│ ○ Create new Product                                                          │
│ ○ Link source only                                                            │
├──────────────────────────────────────────────────────────────────────────────┤
│ ▸ Supporting context: raw provider payload / legacy catalog link              │
├──────────────────────────────────────────────────────────────────────────────┤
│ [Back]                                                        [Apply changes]│
└──────────────────────────────────────────────────────────────────────────────┘
```

### Notes

* The workflow title says `Add Item from Lookup`, not `Create Catalog Item`.
* The destination line says `Product` because Product is the canonical result.
* Raw payload and legacy catalog link are supporting context only.

## Pattern B — Select / Build / Preview / Submit

Use for demand-to-order or multi-demand PO builder workflows.

```text
Stage 1 — Select demand
  Choose demand lines or stock needs.

Stage 2 — Build PO
  Group by vendor and create draft lines.

Stage 3 — Preview impact
  Show demand coverage, planned on-order quantities, cost/retail impact, and blockers.

Stage 4 — Submit
  Convert draft coverage into inbound availability and vendor-facing order state.
```

### Target wireframe

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Build Purchase Order from Demand                         [Save draft] [Exit]│
│ Vendor: Ingram · Store: Main · Canonical source: DemandLine                  │
├──────────────────────────────────────────────────────────────────────────────┤
│ [1 Select demand] → [2 Build PO] → [3 Preview impact] → [4 Submit]          │
├──────────────────────────────────────────────────────────────────────────────┤
│ Stage summary                                                                │
│ 8 demand lines selected · 6 can be ordered · 2 need vendor/source attention. │
├──────────────────────────────────────────────────────────────────────────────┤
│ Demand grouping                                                              │
│ Product / variant             Need  Vendor  Cost  Status        Action      │
│ Left Hand / New Paperback      2     Ingram  5.20  Ready         Include     │
│ Dune / New Hardcover           1     —       —     Needs source  Fix source  │
├──────────────────────────────────────────────────────────────────────────────┤
│ Blockers appear in-line beside affected rows.                                │
├──────────────────────────────────────────────────────────────────────────────┤
│ [Back]                                                    [Create draft PO] │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Notes

* Do not label the flow `Purchase Request` or legacy `TBO` unless preserving an explicit compatibility alias.
* Demand remains the source of need; PO draft coverage is explained as planned coverage until submit.
* Vendor/source problems appear on the rows that need action.

## Pattern C — Receive / Match / Preview / Post

Use for vendor shipment receiving and receipt matching.

```text
Stage 1 — Receive shipment
  Enter vendor, shipment facts, and raw line quantities.

Stage 2 — Match lines
  Match receipt lines to PO lines or mark unmatched.

Stage 3 — Preview impact
  Show accepted quantities, inbound allocation conversion, customer-ready impact, and inventory posting effects.

Stage 4 — Post receipt
  Post accepted inventory and update demand/PO continuity.
```

### Target wireframe

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Receive Vendor Shipment                                  [Save draft] [Exit]│
│ Vendor: Ingram · Receipt: Draft · Store: Main                                │
├──────────────────────────────────────────────────────────────────────────────┤
│ [1 Receive shipment] → [2 Match lines] → [3 Preview impact] → [4 Post]      │
├──────────────────────────────────────────────────────────────────────────────┤
│ Stage summary                                                                │
│ 12 lines entered · 10 matched · 2 unmatched · 4 customer-ready after post.   │
├──────────────────────────────────────────────────────────────────────────────┤
│ Match workpad                                                                │
│ Receipt line             Qty  Suggested PO line      Match state    Action  │
│ Left Hand paperback      2    PO-104 / line 3        Confirmed      Change  │
│ Unknown ISBN             1    —                      Unmatched      Resolve │
├──────────────────────────────────────────────────────────────────────────────┤
│ Preview impact                                                               │
│ Accepted qty posts inventory. Rejected/damaged/wrong/short does not post.   │
│ Customer-ready: Jane Doe, Alex Kim                                           │
├──────────────────────────────────────────────────────────────────────────────┤
│ [Back]                                                     [Post receipt]   │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Notes

* The workpad should show matching status before posting impact.
* Inventory posting language should remain explicit: accepted quantity posts; rejected/damaged/wrong/short does not.
* Customer-ready outcomes should be visible before post, not discovered after post.

## Pattern D — Review / Repair / Confirm

Use for legacy catalog link repair, import provenance repair, and compatibility cleanup.

```text
Stage 1 — Review source
  Show current legacy/import link and canonical Product target.

Stage 2 — Repair link
  Choose product, unlink, or mark no canonical product.

Stage 3 — Confirm
  Preview effect and audit reason.
```

### Target wireframe

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Repair Legacy Catalog Link                              [Cancel]            │
│ Legacy CatalogItem #12345 · Target type: Product                            │
├──────────────────────────────────────────────────────────────────────────────┤
│ [1 Review source] → [2 Choose product] → [3 Confirm]                        │
├──────────────────────────────────────────────────────────────────────────────┤
│ Stage summary                                                                │
│ This is an admin/provenance repair. It does not create inventory or demand. │
├──────────────────────────────────────────────────────────────────────────────┤
│ Current link                                                                 │
│ CatalogItem #12345 → Product #987 The Left Hand of Darkness                 │
│                                                                              │
│ Repair action                                                                │
│ ○ Keep existing link                                                         │
│ ● Link to different Product [Search product...]                              │
│ ○ Unlink legacy record                                                       │
│                                                                              │
│ Audit reason * [ Duplicate import cleanup                         ]         │
├──────────────────────────────────────────────────────────────────────────────┤
│ [Back]                                                       [Confirm repair]│
└──────────────────────────────────────────────────────────────────────────────┘
```

### Notes

* This flow is explicitly admin/provenance repair, not routine product setup.
* It should require a permission and an audit reason.
* It should never appear as the default path for adding a product.

## Pattern E — Intake / Decide / Complete

Use when a workflow has a human decision step, such as buyback refinement or future customer-special-order decisions.

```text
Stage 1 — Intake
  Capture item/customer facts.

Stage 2 — Review / price / source
  Produce recommendation or proposal.

Stage 3 — Decide
  Customer/staff accepts, rejects, substitutes, or defers.

Stage 4 — Complete
  Post inventory, issue credit, create demand, or mark no action as appropriate.
```

### Target wireframe

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Customer Item Decision                                  [Save draft] [Exit] │
│ Customer: Jane Doe · Product: The Left Hand of Darkness                      │
├──────────────────────────────────────────────────────────────────────────────┤
│ [1 Intake] → [2 Review options] → [3 Decide] → [4 Complete]                 │
├──────────────────────────────────────────────────────────────────────────────┤
│ Stage summary                                                                │
│ New copy unavailable. Used copy available on hand. Staff decision required. │
├──────────────────────────────────────────────────────────────────────────────┤
│ Options                                                                      │
│ Option                  Availability       Customer impact       Action      │
│ Order new paperback     Vendor unknown     Wait for sourcing     Source      │
│ Hold used-good copy     On hand now        Pickup today          Select      │
│ Cancel demand           —                  No fulfillment        Select      │
├──────────────────────────────────────────────────────────────────────────────┤
│ [Back]                                                  [Record decision]   │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Notes

* Decision workflows need a visible decision state, not only a status enum.
* Available choices should describe customer impact.
* Completion should be explicit when inventory, stored value, demand, or audit history changes.

## Cross-workflow components

### Stage summary component

```text
Stage summary
- Count of affected lines / records
- Ready vs blocked counts
- What continuing will do
- Highest-severity blocker or warning
```

### Inline blocker component

```text
Blocked: Missing preferred vendor
Why it matters: This line cannot be ordered until a vendor source exists.
Action: [Add vendor source]
```

### Impact preview component

```text
If you continue:
- 2 demand lines will be covered by this PO.
- 4 accepted units will post to inventory.
- 1 customer will become ready for pickup.
- No legacy catalog records will be modified.
```

### Supporting context component

```text
Supporting context
▸ Import/source details
▸ Legacy catalog link
▸ Raw provider payload
▸ Audit history
```

Supporting context is collapsed by default unless the current stage is explicitly a provenance/admin repair workflow.

## Review checklist

Before implementing a multi-stage workflow, answer:

1. What canonical record owns this workflow?
2. What are the staff-language stages?
3. Which stage mutates data, and is that mutation previewed first?
4. Where are blockers shown, and can the user act on them there?
5. What is saved if the user exits before completing?
6. Which context is canonical, and which context is legacy/provenance?
7. What permission gates admin/provenance repair?
8. What audit event or reason is required at completion?
