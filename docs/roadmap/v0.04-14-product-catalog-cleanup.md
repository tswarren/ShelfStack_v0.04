# v0.04-14 — Product-Native Item Layout and Legacy Catalog Quarantine

## Status

**Proposed planning milestone.** This document defines the definitive layout, information architecture, and implementation boundaries for cleaning up the remaining v0.03 catalog vocabulary after v0.04-13.

This is a planning/specification document only. It does not mark the milestone complete.

## Purpose

ShelfStack v0.04 made `Product` the canonical item record, `ProductIdentifier` the canonical identifier layer, and `ProductVariant` the operational grain for selling, inventory, ordering, receiving, demand, allocations, and fulfillment.

The remaining UX problem is that legacy `CatalogItem` concepts still leak into ordinary item setup and import flows. This milestone quarantines legacy catalog/admin surfaces and makes staff-facing item work product-native.

## Guiding decision

```text
Product is the item.
ProductVariant is the operational unit.
Catalog/import records are provenance and legacy administration only.
```

Ordinary staff workflows must not require a user to think in terms of `CatalogItem` as the parent of a product. Legacy catalog context may remain available for import review, historical troubleshooting, buyback/external lookup continuity, and temporary administration, but it must be visually and navigationally secondary.

## Non-goals

This milestone does **not** implement:

* Product groups (`v0.04-3`) or related-edition search.
* A new external-provider architecture.
* Keyword/title/author external lookup expansion.
* Automatic subject/category mapping.
* Automatic pricing updates from external data.
* Vendor-direct fulfillment or carton receiving.
* Phase 9c financial postings.
* A broad visual redesign beyond the item/catalog cleanup scope.

## Domain boundaries

### Canonical records

| Record | Role in the layout |
| --- | --- |
| `Product` | Staff-facing item record and descriptive metadata home. |
| `ProductIdentifier` | ISBN, UPC, GTIN, house, vendor, and legacy freeform identifiers for lookup. |
| `ProductVariant` | Sellable, stockable, reservable, orderable, receivable, buyback, POS, and inventory grain. |
| `DemandLine` / `DemandAllocation` | Customer/store need and supply claims. |
| `SourcingRun` / `SourcingAttempt` / `VendorResponse` | Vendor inquiry and availability workflow. |
| `PurchaseOrder` / `Receipt` | Ordering and receiving documents. |
| `Inventory::Post` / ledger / balance | Authoritative stock mutation and reporting source. |

### Quarantined records

| Record | Allowed role |
| --- | --- |
| `CatalogItem` | Retain-temporary bibliographic/admin shell for historical import, external lookup, and troubleshooting. |
| Legacy catalog identifiers | Migration/import reference only; do not expose as the ordinary identifier model. |
| `products.catalog_item_id` | Optional legacy/provenance link only; not a normal product setup field. |

## Information architecture

### Top-level navigation

```text
Items
├── Product search / index
├── Add item
├── External lookup / import review
└── Legacy catalog admin (secondary/admin-only)
```

The primary Items workspace should route staff toward products, variants, demand, inventory, ordering, and receiving. Legacy catalog administration should not appear as a peer to canonical product work unless the user has explicit admin/import permissions.

`catalog_item_id` must not be shown as a normal editable setup field. If a product has a legacy catalog link, show it as read-only provenance in the collapsed Legacy / Import Context section. If an admin must repair a link, that should be an explicit admin action, not part of routine product setup.

### External lookup / import review

```text
External Lookup
├── Search / scan input
├── Local match panel
│   ├── Product matches
│   ├── Identifier matches
│   └── Variant matches
│
├── External candidate panel
│   ├── Bibliographic/source metadata
│   ├── Provider/source confidence
│   └── Raw payload access, permission-gated
│
├── Import decision
│   ├── Create new Product
│   ├── Fill blanks on existing Product
│   ├── Link as source/provenance
│   └── Ignore / no action
│
└── Post-import destination
    ├── Product detail
    ├── Variant setup
    └── Identifier review
```

External lookup should produce or update canonical product records. It should not direct ordinary users into a legacy CatalogItem-first workflow.

### Legacy catalog admin

```text
Legacy Catalog Admin
├── Search legacy records
├── View linked product, if any
├── View import/source history
├── Repair source/provenance links, permission-gated
└── Raw payload / troubleshooting details
```

Legacy catalog admin is a temporary maintenance surface. It should be visually labeled as legacy/import administration and should not be used for normal product setup.

## Screen-by-screen changes

| Screen / area | Direction |
| --- | --- |
| Items index | Product-first search and operational summary. Do not present CatalogItem as the primary result type. |
| Product detail | Product header, operational summary, work tabs, secondary import/legacy context. |
| Product new/edit | Remove normal `catalog_item_id` picker; show legacy link only as read-only provenance or admin repair. |
| Product identifiers | Manage on `product_identifiers`; legacy identifiers appear only as migrated/freeform provenance when needed. |
| Variant operations drawer | Keep demand, inventory, ordering, and receiving product/variant-native; remove legacy activity once no historical data path exists. |
| External lookup | Resolve local Product/ProductIdentifier matches first; create/update Product records. |
| Ingram/import workflows | Report Product/ProductVariant outcomes first; keep CatalogItem only as retained source/admin detail if still required. |
| Buyback intake | Resolve by ProductIdentifier/ProductVariant lookup; legacy CatalogItem fallback only for temporary compatibility. |
| POS line lookup | ProductIdentifier and variant lookup code only; no legacy catalog dependency. |
| Reports | Continue reporting by product variant, department/subdepartment, tender, demand, inventory, and operational ledgers. |

## Primary action rules

Use the existing view-contract direction:

* One visually obvious primary action per page or section.
* Preserve context with drawers, modals, and Turbo updates instead of unnecessary full page hops.
* Summarize related records before exposing full detail tables.

Recommended primary actions:

| Context | Primary action |
| --- | --- |
| Product with no variants | Create variant |
| Product with active variants | Add demand / order / sell depending entry context |
| External candidate with no local match | Create Product |
| External candidate with local product match | Review and fill blanks |
| Legacy catalog record with linked product | Open Product |
| Legacy catalog record without linked product | Create/link Product, admin-only |

## Copy and terminology rules

Use:

* Item
* Product
* Variant
* Identifier
* External lookup
* Import source
* Legacy catalog admin
* Product metadata
* Variant operations

Avoid in ordinary workflows:

* Catalog item as the parent object
* Catalog-linked product
* Create catalog item
* Catalog identifier
* Purchase request
* TBO, except where retained as deprecated compatibility or staff-facing manual store-demand language is explicitly approved

## Implementation slices

### Slice A — Audit and routing map

* Inventory every route, controller, presenter, view, helper, and service that exposes `catalog_item_id`, `CatalogItem`, legacy catalog identifiers, or catalog-first UI copy.
* Classify each use as canonical removal, provenance display, import/admin quarantine, or temporary compatibility.
* Produce a before/after routing map before modifying behavior.

### Slice B — Product form and detail layout

* Remove routine `catalog_item_id` editing from product new/edit forms.
* Add read-only Legacy / Import Context where a legacy link exists.
* Reorganize product detail into header, operational summary, tabs, and secondary context.
* Ensure ProductIdentifier management is the visible identifier workflow.

### Slice C — External lookup/import destination cleanup

* Make external lookup review resolve local products and identifiers first.
* Make import actions create/update Product and ProductIdentifier records as the primary outcome.
* Move raw payload and legacy source details behind permission-gated secondary UI.
* Update success messages and redirects to product-native destinations.

### Slice D — Legacy catalog quarantine

* Move legacy catalog admin to a secondary/admin-only navigation path.
* Label it explicitly as legacy/import administration.
* Hide or remove ordinary navigation that implies CatalogItem is the canonical item record.
* Keep compatibility redirects only where necessary.

### Slice E — Variant and operational cleanup

* Remove legacy activity sections once no historical row path exists.
* Verify buyback, POS, demand, ordering, receiving, and reports remain Product/ProductVariant native.
* Rename remaining user-facing legacy labels to product/demand-native terms.

### Slice F — Documentation and verifier

* Update `docs/schema-reference.md`, `docs/domain-model.md`, and relevant roadmap docs with the final quarantine rule.
* Add a lightweight verifier that fails on disallowed user-facing strings or routine form fields where practical.
* Document allowed exceptions for import/admin compatibility.

## Acceptance criteria

1. Staff can create and edit a Product without seeing `catalog_item_id` as a normal setup field.
2. Product detail pages present Product as the canonical item and ProductVariant as the operational unit.
3. Product identifiers are managed through `product_identifiers`, not through legacy catalog identifier UI.
4. External lookup/import actions land on Product/ProductVariant outcomes by default.
5. Legacy CatalogItem records are accessible only through clearly labeled legacy/import admin surfaces or collapsed provenance sections.
6. POS, buyback, demand, ordering, receiving, and inventory paths do not require CatalogItem as an operational dependency.
7. Remaining compatibility shims are documented with owners and removal criteria.
8. User-facing copy avoids CatalogItem as the normal parent concept.
9. The app remains runnable after each slice.
10. v0.04 invariants remain unchanged.
